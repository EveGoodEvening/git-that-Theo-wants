# C10 pain-point mapping

This mapping connects the six README pain points to the prototype features and implementation chunks demonstrated by `examples/demo.ts`. The demo is a local, deterministic, in-process simulation of the data model; it does not model production/server enforcement.

| # | README pain point | Demonstrated in `examples/demo.ts` | Prototype feature(s) | Chunk(s) |
|---:|---|---|---|---|
| 1 | Commit a `.env` file without exposing plaintext to every repository reader. | Adds `.env` to a snapshot as a `secret-blob`, checks raw stored bytes do not contain the plaintext, and verifies only the local demo key recovers it. | Content-object envelope, in-memory store, first-class encrypted secret blobs, file visibility keeping `.env` out of public export. | C1, C2, C5, C6, C10 |
| 2 | Fine-grained permissions: private files, private PR-equivalents, publish timing, and unpublish. | Keeps `notes/private-roadmap.md` readable through the owner-side virtual tree while omitting it from the public bundle; starts the PR-equivalent snapshot private, then runs `publish` and `unpublish`. | Per-file and per-snapshot visibility states, deterministic visibility matrix, append-only publish/unpublish events, public/private manifest split. | C4, C5, C6, C9, C10 |
| 3 | Open source should allow a public subset without forcing every file and in-flight change public. | Builds a public bundle containing only `src/public.ts`; `.env`, the private roadmap path, and their object ids are absent from public entries and bundle objects. | Public projection ids derived only from public entries, public export bundle with public objects only, private manifest for owner-side correlation. | C6, C8, C9, C10 |
| 4 | Commits and branches are a poor fit compared with snapshot + tag workflows. | Creates a content-addressed snapshot, tags it as `demo-v1`, records the tag move in the op-log, and attaches manifest refs without changing the `SnapshotId`. | Snapshot core identity, bookmark/tag pointers, op-log, manifest-ref attachment excluded from snapshot identity. | C4, C6, C9, C10 |
| 5 | Worktrees should not hijack or lock each other. | Creates two workspaces checked out at the same `main` ref; both snapshot independent changes while `main` remains pointed at the shared base. | Workspace ids independent of ref ownership, no checkout lock, dirty working copy snapshots, conflict-as-data foundation. | C4, C7, C9, C10 |
| 6 | Source control should not depend on cloning real OS/filesystem trees. | Runs the whole flow with `MemoryStore`, `VirtualTree`, and the in-memory public export bundle; no clone/materialization path is needed for the demo. | Pluggable store with in-memory backend, virtual filesystem operations, public bundle export path, optional real-FS adapter only after C6 filtering. | C2, C3, C6, C8, C9, C10 |
